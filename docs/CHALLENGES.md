# Проблеми, які виникали — і як ми їх вирішували

Це чесний інженерний лог: реальні граблі побудови **локальної** (host-GPU + контейнери) системи на Windows із 4GB GPU. Кожен пункт — *проблема → діагноз → фікс → урок*.

---

### 1. `ffmpeg` не на PATH процесу → `WinError 2`
**Проблема.** STT-воркер падав на `whisperx.load_audio` із «The system cannot find the file specified».
**Діагноз.** WhisperX/pyannote шелять у голий `ffmpeg`. Бінарник лежав у `.venv\Scripts\ffmpeg.exe`, але **прямий** запуск `python.exe scripts\worker.py` (без активації venv) НЕ додає `Scripts` у PATH. Бенчмарк працював лише тому, що його запускали з активованим venv.
**Фікс.** У `scripts/_env.py` функція `_ensure_venv_bin_on_path()` додає каталог інтерпретатора в PATH на імпорті — STT працює незалежно від оболонки (раннер, cron, сервіс).
**Урок.** Не покладайся на PATH оболонки для субпроцесів; роби середовище самодостатнім.

### 2. Нативний Postgres затіняв порт 5432
**Проблема.** Host-воркер не міг підключитись до Postgres: `FATAL ... role "cairnwise" ...`, хоча в контейнері все працювало.
**Діагноз.** На `127.0.0.1:5432` слухав **нативний PostgreSQL 18** (Windows-служба), що затіняв docker-Postgres для host-клієнтів. API цього не помічав — він ходить у `postgres:5432` внутрішньою docker-мережею.
**Фікс.** Зсунули host-порт docker-Postgres на **5433** (як уже зробили з Redis `6380` і Qdrant `6543`, бо їх 6379/6333 займали контейнери інших проєктів). `.env` на хості → `localhost:5433`.
**Урок.** «Працює в контейнері» ≠ «працює з хоста». Внутрішня мережа й експонований порт — різні площини.

### 3. OOM на 4GB GPU (medium/large-v3 не влазять)
**Проблема.** Whisper `medium`/`large-v3` падали з CUDA OOM на кроці VAD — навіть зі зменшеним batch.
**Діагноз.** Памʼять VAD масштабується з довжиною аудіо (зустрічі 30-50 хв); OOM стається ще ДО ASR-батчингу, тож `batch_size` не рятує. Каскад: large-v3 падає в процесі → засмічує VRAM → наступні моделі теж OOM.
**Фікс.** Зафіксували `small` як стек (бенчмарком підтверджено прийнятну якість на укр.); запускаємо найбезпечнішу модель першою; звільняємо модель перед align.
**Урок.** Обмеження заліза — це проєктне рішення, а не баг. Документуй межу й обирай стек під неї.

### 4. Завантажені файли невидимі host-воркеру (named volume)
**Проблема.** Воркер не знаходив аудіо зустрічі, хоча API його прийняв.
**Діагноз.** API (контейнер) зберігав файли в Docker **named volume** `uploads` — host-процес його не бачить.
**Фікс.** Перевели на **bind-mount** `./data/uploads:/data/uploads` — і контейнер, і host бачать той самий каталог. Воркер реконструює host-шлях за розкладкою `<storage>/<project_id>/<файл>`.
**Урок.** Гібрид «контейнер-API + host-воркер» вимагає спільної ФС через bind-mount, а не named volume.

### 5. `create_all` не розширює нативний enum
**Проблема.** Після додавання статусів `ingesting`/`ingested` інжестія мала б падати на записі статусу.
**Діагноз.** SQLAlchemy зберігає `MeetingStatus` як **нативний Postgres enum** `meetingstatus`. `create_all` створює нові *таблиці*, але НЕ розширює існуючий enum-тип — у БД лишались старі 4 значення.
**Фікс.** Легка ідемпотентна міграція в `db.py`: `ALTER TYPE meetingstatus ADD VALUE IF NOT EXISTS 'ingesting' BEFORE 'failed'` — в **autocommit** (бо `ADD VALUE` не можна використати в тій самій транзакції).
**Урок.** До Alembic — тримай явний список ідемпотентних DDL-міграцій; enum-розширення легко проґавити.

### 6. Echo-баг у промпті резюме
**Проблема.** LLM повертала поле `summary` як дослівний текст-**опис** зі схеми («2-4 речення українською…»), а `confidence` = 0.0.
**Діагноз.** Ми поклали інструкції *як приклади-значення* всередину JSON-схеми — модель їх «відлунила».
**Фікс.** Розділили **опис полів** і **приклад структури** (вигадана інша зустріч) + явне «заповни реально, не копіюй опис». Додали авто-аудит цитат (будь-яке `[#N]` поза діапазоном = галюциноване посилання).
**Урок.** У промпті ніколи не став інструкцію туди, де очікуєш згенероване значення.

### 7. `meeteval` (канонічний cpWER) не збирається на Windows
**Проблема.** `pip install` усього падав через `meeteval` (потребує MSVC Build Tools, немає wheel).
**Фікс.** Власна pure-Python реалізація cpWER: оптимальне співставлення спікерів (Hungarian, `scipy.linear_sum_assignment`) + `jiwer`. Метрика працює без C++ тулчейну.
**Урок.** Одна крихка залежність може блокувати весь install — ізолюй або заміни.

### 8. Тонкий образ API (torch-free) при host-ML
**Проблема.** Якщо тягнути torch/whisperX/bge-m3 в образ API — він роздувається до гігабайтів.
**Фікс.** Чіткий шов: `app/rag/{chunker,vector_store,schema,entities}` — **torch-free** (імпортовні в CPU-контейнері для read-path), а `app/rag/embedder` (torch+FlagEmbedding) імпортується **лише ліниво** на хості. Образ API лишається ~250 МБ.
**Урок.** Проводь межу torch/не-torch на рівні модулів — це дозволяє одному пакету жити і в контейнері, і на хості.

### 9. Колізія VRAM на інжесті: bge-m3 + neural-chat не вміщаються разом у 4GB
**Проблема.** Перший наскрізний прогін `ingest.py` падав на витяганні сутностей: `HTTPError 500` від Ollama — embedding (48 чанків) уже відпрацював, але крок LLM валився.
**Діагноз.** `ingest_meeting()` тримає `Embedder` (bge-m3, ~2 GB VRAM) резидентним на весь run, а далі в тому ж процесі кличе Ollama, який намагається підняти `neural-chat` (Q4_0, ~4 GB) на **той самий 4 GB GPU** → сумарно > 4 GB → Ollama повертає 500. Перевірено ізольовано: з вивантаженим bge-m3 той самий промпт (`num_ctx=8192`, ~15k символів) відпрацьовує штатно (3.6 GB VRAM, валідний JSON). Тобто причина — **co-residency двох моделей**, а не розмір контексту.
**Фікс.** На 4 GB одна з моделей має звільнити GPU: або стартувати інжест із чистого GPU (тоді Ollama офлоудить частину шарів на CPU й вкладається — наскрізний прогін: 48 чанків → 3 action-items + 10 decisions, 474 с), або embed на CPU (`--device cpu`), лишивши GPU під LLM. Той самий принцип «не тримати дві моделі на GPU одночасно», що й для STT↔LLM.
**Урок.** На малому GPU embed-крок інжесту й LLM-крок — це теж пара, що конкурує за VRAM; на проді ingest-воркер має або тримати embedder на CPU, або явно звільняти VRAM перед викликом LLM.

### 10. Запис у браузері: крос-браузерні пастки (mic+system audio)
**Проблема.** Перша версія рекордера падала на старті із загальним «не вдалося почати запис», а адверсаріальне ревʼю (воркфлоу з 5 агентів) знайшло ще 4 приховані дефекти.
**Діагноз + фікси.** (1) **Порядок захоплення:** `getDisplayMedia` суворо вимагає свіжої user-activation — кличемо його ПЕРШИМ, до `await getUserMedia`. (2) **Safari:** `MediaRecorder` не вміє webm → додали mp4/aac-фолбек і стампуємо Blob **реальним** `recorder.mimeType`, а не припущенням (інакше mp4-байти йшли як `.webm`); `AudioContext` → з фолбеком на `webkitAudioContext`. (3) **Обрив «Stop sharing»** під час запису — `'ended'`-слухач на системному треку → тост, далі лише мікрофон. (4) **0-байтний запис** більше не вантажиться; **подвійний клік «Зупинити»** не вішає проміс (memoized stop + очищення ref до await). (5) Помилка показує **реальний `DOMException.name`** у дужках для діагностики.
**Урок.** Браузерні media-API крихкі й різні між Chrome/Safari/Firefox; порядок виклику, реальний тип кодека і життєвий цикл треків — усе має значення. Адверсаріальне ревʼю кодом, який не запустиш headless, окупається.

### 11. «Застрягло в черзі» = host-воркер не запущено
**Проблема.** Зустріч лишалась «У черзі», натискання «Транскрибувати» нічого не робило (хіба множило задачі в Redis).
**Діагноз.** Транскрипція/резюме виконують **host-воркери** (`worker.py`, `summary_worker.py`), а API лише `LPUSH` у чергу. Воркери не були запущені — `BRPOP` нікому робити, задачі копились.
**Фікс/урок.** Тримати обидва воркери запущеними (README, крок 4) — це навмисний поділ (API torch-free, GPU на хості). Окремо: одна зустріч мала **відсутній на диску аудіофайл** (osиротілий рядок) → `FileNotFoundError`; прибрали рядок. У проді бракує health-сигналу «воркер живий» + UI-підказки, коли черга не рухається — у дорожній карті.
