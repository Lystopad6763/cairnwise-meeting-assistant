# Аналіз результатів Extraction Agent

В оригінальному ДЗ було порівняти дві моделі — Ollama (будь-яку локальну) і OpenAI. Було цікаво порівняти різні моделі від різних організацій, тому розширив список до **6 моделей**:

| Модель              | Автор / Організація | Тип               |
| ------------------------- | ----------------------------------- | -------------------- |
| neural-chat:latest        | Intel                               | Self-hosted (Ollama) |
| phi3:latest               | Microsoft                           | Self-hosted (Ollama) |
| llama2:latest             | Meta (Meta AI)                      | Self-hosted (Ollama) |
| mistral:latest            | Mistral AI                          | Self-hosted (Ollama) |
| gpt-4o-mini               | OpenAI                              | Cloud API            |
| claude-haiku-4-5-20251001 | Anthropic                           | Cloud API            |

Тестував на трьох типах тексту:

- **simple** — простий структурований протокол зустрічі (3 завдання)
- **chaotic** — хаотична зустріч де всі перебивають (4 завдання)
- **technical** — технічний сінк з технічною термінологією (5 завдань)

Всього **18 запусків** (6 моделей × 3 датасети).

---

## 1. Обмеження експерименту (залізо)

Усі latency які я виміряв — це результат на **моєму ПК** (GTX 1050 4GB + 16GB RAM). На іншому обладнанні ті самі моделі працювали б зовсім по-іншому.

Моделі (mistral, llama2, neural-chat) важать 4+GB і **не влазять у мою VRAM** (всього 4GB). Тому Ollama використовує RAM + CPU як fallback — це в 5-10 разів повільніше ніж робота на GPU.

**Що залежить від заліза:**

- Latency локальних моделей (Ollama)

**Що НЕ залежить від заліза (валідно на будь-якому ПК):**

- Latency хмарних моделей — залежить тільки від інтернету і серверів
- JSON валідність, галюцинації, якість українською — це властивості моделі

Тому всі мої висновки про **якість моделей** справедливі на будь-якому залізі. А ось вердикт "локальні моделі повільні" — справедливий тільки для мого ПК.

# Найцікавіші спостереження з результатів

## 2. Хмарні моделі вигадали рік

Сьогодні **03.05.2026**. У моїх текстах зустрічі року взагалі не було — просто "10 травня", "15 травня". Але хмарні моделі не змогли стриматись і **вигадали роки**:

| Модель | Датасет | Що вигадала                                                          |
| ------------ | -------------- | ------------------------------------------------------------------------------ |
| GPT-4o-mini  | chaotic        | `2023-05-05`, `2023-05-11`, `2023-05-13`, `2023-05-12`                |
| GPT-4o-mini  | technical      | `2023-05-14`, `2023-05-11`, `2023-05-13`, `2023-05-16`, `2023-05-18` |
| Claude Haiku | technical      | `2024-05-14`, `2024-05-11`, `2024-05-13`, `2024-05-16`, `2024-05-18` |

Найцікавіше — **жодна не вгадала 2026**:

- GPT поставив **2023** (на три роки в минуле)
- Claude поставив **2024** (на два роки в минуле)

Ймовірно що моделі "застрягли" в часі коли їх тренували. Вони тренувались на даних до 2023-2024 року, тому "поточний" рік для них саме такий. Це класична ілюстрація галюцинації — модель не знає чогось, але впевнено вигадує відповідь замість того щоб написати `null`.

**!!! Тобто при роботі з даними краще завжди передавати точні дати в промпт.**

Локальні моделі (mistral, neural-chat) виявились чеснішими — вони залишили формат "10 травня" як в оригіналі і не вигадували року.

## 3. llama2 натренована на англомовних даних.

Я подавав на вхід **український текст**. llama2 у відповідь:

- Перекладає імена: **Мартин → Martin**, **Оля → Olga**
- Пише summary англійською: `"Meeting Minutes"`, `"Meeting notes"`
- Decisions англійською: `"Use React for frontend"`, `"deploy on AWS"`

Найгірше — на хаотичному тексті llama2 приписала завдання **"Anna"** хоча в оригіналі його робив Іван. Тобто не просто переклала ім'я Це серйозна галюцинація.

## 4. Коли використовувати Ollama (self-hosted)

### 4.1 Що дає Ollama

- **Приватність** — дані не виходять за межі твого комп'ютера
- **Безкоштовно** за запит, але не за GPU
- **Офлайн** — інтернет не потрібен взагалі

### 4.2 Коли це реально треба

- Коли є строга вимога щодо PII.
- Коли серверу заборонений вихід в інтернет.

### 4.3 Найкраща локальна модель за моїми тестами — neural-chat

З 3 запусків neural-chat (по одному на кожен датасет):

- Валідних JSON: **3/3 датасети** ✅
- Знайдено завдань: **12/12 (100%)** ✅
- Галюцинацій: **0** ✅

---

## 5. Коли використовувати OpenAI або Claude

### 5.1 Що дає хмара

| Параметр                                    | Хмара                          | Ollama                                         |
| --------------------------------------------------- | ----------------------------------- | ---------------------------------------------- |
| Latency                                             | **3-7с**                     | 35-110с (особисто на мому ПК) |
| Стабільність JSON                       | **100% (6/6)**                | 58% (7/12)                                     |
| Українська мова                       | **Тримає**              | llama2 ламає                              |
| Робота на хаотичному тексті | **Знаходить все** | Більшість пропускають      |

### 5.2 Коли викорситовувати хмару

- **Real-time системи** — чат-боти, асистенти де очікування 30 секунд неприпустиме
- **Критичні задачі** — юридичні, медичні документи де помилка коштує дорого
- **Хаотичний input** — розмовна мова, інтерв'ю, транскрипти зустрічей
- **Технічний жаргон** — спеціалізована термінологія

## 6. Гібридний підхід

Після експерименту я зрозумів що **не треба вибирати** одну сторону. Найрозумніше — комбінувати.

### 6.1 Ідея: Ollama → fallback → Claude

```python
def extract_with_fallback(text: str) -> dict:
    # Спочатку пробуємо локальну neural-chat — безкоштовно і надійно
    result = extract_meeting_data(text, provider="ollama", model="neural-chat")
  
    # Якщо JSON зламаний — переходимо на Claude
    if not result.get("_valid_json"):
        print("Локальна модель не впоралась, переходимо на Claude...")
        result = extract_meeting_data(text, provider="claude")
  
    return result
```

### 6.2 Економія

З моїх результатів **neural-chat впоралась з усіма 3 датасетами без єдиного fallback**. Якщо взяти всі 4 локальні моделі разом — 7 з 12 валідних JSON, тобто майже половина потрапила б на хмару. І це ОК — все одно платимо лише за частину.

### 6.3 Ensemble (наступний крок)

Запускати 2 моделі **паралельно** і дивитись чи їх відповіді збігаються:

- **Збігаються** → результат точно правильний, зберігаємо
- **Різняться** → відправляємо на перевірку людині

Це фактично робить систему саму себе перевіряючою.

## 7. Розширення агента

### 7.1 Підтримка мов

Проблема яку я побачив — **llama2 відповідала англійською** на український текст. Виправлення тривіальне — додати інструкцію в промпт:

```python
def build_prompt(text: str, output_language: str = "українська") -> str:
    return f"""Прочитай текст зустрічі та витягни структуровану інформацію.
Відповідай ТІЛЬКИ мовою: {output_language}

Текст:
{text}
..."""
```

Для перекладу між мовами (наприклад український input → English JSON):

1. Перекласти input на англійську (більшість моделей краще розуміють англійський input)
2. Запустити агента з `output_language="English"`
3. Готовий результат англійською

### 7.2 confidence_score

Ідея — модель сама оцінює свою впевненість в кожному завданні (від 0 до 1). Це робиться додаванням поля в промпт:

```python
"""Поверни ТІЛЬКИ валідний JSON:
{
  "tasks": [
    {
      "owner": "ім'я",
      "task": "що зробити",
      "deadline": "дата або null",
      "confidence": 0.95
    }
  ]
}

Для confidence:
1.0 = точно знаю з тексту
0.5 = є сумніви  
0.0 = вгадав"""
```

### 7.3 Як це використовувати в production

| confidence | Дія                                                                |
| ---------- | --------------------------------------------------------------------- |
| > 0.9      | Зберігаємо автоматично                           |
| 0.5–0.8   | Відправляємо на перевірку модератору |
| < 0.5      | Відхиляємо, просимо повторний запис    |

Це значно зменшує ризик що зламана відповідь моделі потрапить далі по pipeline.

| Місце | Модель           | Чим сподобалась                                            | Що не так                                                                          |
| ---------- | ---------------------- | ------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------- |
| 1          | **claude-haiku** | Найшвидший (3с), 0 галюцинацій на simple/chaotic | Вигадує рік 2024 на technical                                                 |
| 2          | **neural-chat**  | Єдина локальна з 100% знайдених завдань    | Повільна на technical (65с)                                                    |
| 3          | **gpt-4o-mini**  | Стабільний JSON через `json_object` mode                | Вигадує рік 2023, дорожча за Claude                                    |
| 4          | **mistral**      | Тримає українську, добре на simple/technical      | Зламала JSON на chaotic                                                          |
| 5          | **llama2**       | Знаходить завдання на простому тексті   | Перекладає все на англійську, ламає JSON на technical     |
| 6          | **phi3**         | Найменша (2.2GB)                                                 | Найбільше галюцинацій, найгірша JSON-стабільність |

* **Для більшості завдань** → Cloud API (Claude Haiku або GPT-4o-mini). У наших тестах хмарні моделі дали 100% валідних JSON (6/6 запусків), 3-7с latency, знайшли всі завдання навіть на хаотичному тексті. Але обидві моделі вигадали рік (Claude — 2024, GPT — 2023, замість поточного 2026). Тому в production треба **завжди передавати точну дату в промпт явно** + використовувати **LLM as a Judge** для перевірки виводу (з лекції:  *"завжди вставляйте LLM as a Judge, навіть коли робите детерміністичний evaluation"* ).
* **Для масової обробки** (тисячі документів на день, регуляторні вимоги, приватність даних) → Self-hosted (neural-chat від Intel)  **за умови що є власний сервер з потужним GPU** . Знайшов 12/12 завдань, 0 галюцинацій, $0 за запит.
* **Для балансу якість/вартість** → Гібридний підхід: спочатку neural-chat локально, якщо JSON зламаний → fallback на Claude.
* **Для української мови** → НЕ використовувати llama2.
